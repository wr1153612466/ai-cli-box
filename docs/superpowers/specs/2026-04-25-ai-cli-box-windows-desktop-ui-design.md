# AI CLI Box Windows Desktop UI Design

Date: 2026-04-25
Status: approved initial UI concept
Figma: https://www.figma.com/design/SHxadGXK06XF11kg1EpN2H

## Goal

Build the first practical Windows desktop version of AI CLI Box: a local-first workbench for running and managing multiple AI CLI terminal sessions across projects.

The first version should feel like a focused desktop tool, not a web dashboard. It should make repeated AI CLI work faster by keeping session creation, project selection, terminal state, and tool configuration in one application.

## Product Scope

The first practical release supports:

- Multiple AI CLI sessions in tabs.
- Project-grouped session list in the left sidebar.
- New session flow with AI CLI tool, project path, terminal shell, model, and launch profile.
- Tool configuration for Codex, Claude, Gemini, and OpenCode.
- Terminal preferences for font, theme, cursor style, line height, and scrollback.
- Network proxy settings.
- Data management basics.
- Local import/export sync as the first sync mechanism.
- Simplified Chinese as the default UI language and English as a supported language.

The first release should not include full cloud sync, billing, team accounts, or marketplace-style extension management.

## Visual Direction

The UI uses an original dark Windows desktop workbench style:

- Graphite dark surfaces with clear panel separation.
- Tool-specific accent colors only where they identify a session or tool.
- Dense but readable layout for repeated terminal work.
- Compact controls that do not compete with the terminal.
- Chinese-first text sizing with enough room for English strings.

The workflow can be informed by AICoder screenshots, but the app must keep its own visual identity. Do not copy AICoder branding, exact iconography, spacing, or component styling.

## Main Workspace

The main workspace has four areas:

- Top chrome: app title, language, theme, settings, and window controls.
- Left sidebar: session management only.
- Tab strip: open AI CLI sessions.
- Terminal workspace: active terminal and command input.
- Bottom status bar: model and usage status on the left, global tool actions on the right.

### Left Sidebar

The sidebar is responsible only for session work:

- Search sessions.
- Create a new session.
- Open recent/history view.
- Delete closed or selected session when applicable.
- Show project groups.
- Show sessions under each project with running status and relative time.

The sidebar must not contain Codex, Claude, Gemini, or OpenCode configuration navigation. Those tools are selected in New Session and configured in Settings.

### Top Right Controls

Top right controls include:

- Language switcher.
- Theme switcher.
- Settings button.
- Minimize.
- Maximize or restore.
- Close.

The settings button is icon-only. A tooltip may appear on hover, but the text `设置` is not permanently visible in the top chrome.

### Bottom Status Bar

The bottom bar shows current session state and global actions:

- Current model, for example `gpt-5.5 high`.
- Context usage, for example `上下文 100%`.
- Time or quota status, for example `5h 91%`.
- Weekly usage, for example `本周 66%`.
- Global actions on the right: `指令`, `任务队列`, `仪表盘`.

The bottom bar does not show trial labels such as `试用 28天`, and it does not show an `Auto` label in the first design.

## New Session

The New Session flow creates a session from these inputs:

- Session name, optional and auto-generated by default.
- AI CLI tool: Codex, Claude, Gemini, or OpenCode.
- Project path, with browse and recent project shortcuts.
- Terminal shell: PowerShell 7, Windows PowerShell, cmd, or custom shell.
- Model selection when supported by the tool.
- Launch profile, such as balanced, approval-required, sandbox write, autonomous, or custom.
- Optional session template, such as code review, bug fix, feature development, refactor, or test writing.

AI CLI tool selection belongs here rather than in the main sidebar.

## Tool Settings

Settings contains one tool configuration section per supported AI CLI:

- Codex.
- Claude.
- Gemini.
- OpenCode.

Each tool page can include:

- Authentication status.
- CLI executable detection.
- Custom executable path.
- Installed version and reinstall/update actions.
- Launch profiles.
- Tool-specific MCP server configuration if supported.
- Proxy enablement for the tool.

Secrets must be stored securely using a local encrypted storage mechanism appropriate for Windows.

## Terminal Settings

Terminal settings include:

- Default shell.
- Shell detection status and executable path.
- Font family.
- Font size.
- Color theme.
- Cursor style.
- Cursor blink.
- Line height.
- Letter spacing.
- Scrollback line count.

Default shell options:

- PowerShell 7 via `pwsh.exe`.
- Windows PowerShell via `powershell.exe`.
- cmd via `cmd.exe`.
- Custom shell path.

Each session can override the default shell.

## General Settings

General settings include:

- Theme mode: dark, light, or system.
- UI language: Simplified Chinese, English, or system.
- Close-window behavior.
- Notification toggle for completed CLI responses.
- Notification threshold.
- Default project path.
- Sidebar session display count.
- Monthly budget warning placeholder for future use.

## Network Settings

Network settings include:

- Proxy host.
- Proxy port.
- Optional proxy username and password.
- NO_PROXY patterns.
- Test proxy connection action.

Each AI CLI tool can choose whether to use the configured proxy.

## Data And Sync

Data management includes:

- Display local database path.
- Display database size.
- Clean closed sessions.
- Copy logs.
- Run CLI diagnostics.

Sync first supports:

- Export settings to file.
- Import settings from file.
- Optional encryption password for export.

WebDAV sync can be added later after local import/export is stable.

## Localization

The default language is Simplified Chinese.

The first release also supports English. All visible UI strings must come from i18n resource files instead of being hardcoded in components.

Implementation rules:

- Use stable translation keys.
- Keep tool names such as Codex, Claude, Gemini, OpenCode, PowerShell 7, Windows PowerShell, and cmd as product names.
- Reserve enough width for English labels in buttons, tabs, and status bars.
- Localize empty states, validation messages, errors, and settings descriptions.

## States And Error Handling

The UI should represent these states:

- CLI executable detected.
- CLI executable missing.
- Authenticated.
- Not authenticated.
- Session running.
- Session stopped.
- Session failed to start.
- Shell missing.
- Project path missing.
- Proxy test success or failure.
- Secret save failure.

Errors should be specific and actionable. For example, if PowerShell 7 is selected but `pwsh.exe` is not found, the user should see a clear message and a path selection action.

## Implementation Implications

The UI design implies these core technical modules:

- Desktop shell using Tauri and React.
- Terminal rendering using xterm.js.
- Terminal integration using a Rust pseudo-terminal layer.
- Process/session manager.
- Tool profile registry.
- Shell detection service.
- Zustand metadata state store.
- SQLite local session/settings metadata database.
- Secure secret storage through system keyring APIs.
- i18n resource loader.

Terminal output must stream directly from the Rust PTY event bridge into xterm.js. It should not be stored in React component state or Zustand because long AI CLI output would cause unnecessary re-rendering, memory pressure, and degraded terminal responsiveness.

These modules should be planned before implementation begins.

## Open Follow-Up

The UI concept is approved as an initial direction. Future changes can happen during implementation when real terminal behavior, settings persistence, and packaging constraints are clearer.
